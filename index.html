01 · PROBLEM

요청자와 제작자 모두가 같은 지점에서 손실을 겪었습니다.

  • 요청자는 작성 기준을 몰라 입력 부담이 컸습니다.
  • 제작자는 필수 정보가 없어 추가 질문을 반복했습니다.
  • 수정 의견이 여러 경로로 전달돼 이력이 분산됐습니다.
02 · REFRAME

‘원고 생성 속도’가 아니라 ‘입력 품질’을 핵심 문제로 재정의했습니다.

  • AI 초안이 빨라도 입력값이 불완전하면 검수 비용은 남습니다.
  • 업무 유형별로 필요한 질문만 보여주는 동적 폼이 필요했습니다.
  • 한 번 입력한 데이터가 다음 단계로 이어져야 했습니다.
03 · DECISION

전체 자동화보다 규칙을 정의하기 쉬운 3단계를 먼저 선택했습니다.

  • 동적 요청 폼
  • 요청 데이터 구조화
  • 누락 정보 확인 및 요청서 분석
04 · OUTCOME DESIGN

기능 완료가 아니라 업무 변화로 성공 기준을 정의했습니다.

  • 작성 시간과 정보 누락률 감소
  • 추가 질문 및 평균 수정 횟수 감소
  • 초안 제작 시간과 전체 리드타임 단축

06 / FAQ

프로젝트 핵심 질문

문제 정의, 우선순위, 검증 방식과 제 기여 범위를 중심으로 정리했습니다.

01처음 받은 요청을 어떻게 다시 정의했나요?+
처음 받은 요청은 GPT로 원고를 빠르게 만드는 것이었습니다. 하지만 업무를 관찰한 뒤 병목을 원고 작성이 아니라 입력 정보의 불완전성과 단계 간 단절로 재정의했습니다. 해결 범위도 생성 기능 하나가 아니라 동적 폼, 데이터 구조화, 누락 확인으로 바꿨습니다.
02효과는 어떤 지표로 검증하도록 설계했나요?+
요청서 작성 시간, 정보 누락률, 추가 질문 수, 초안 제작 시간, 평균 수정 횟수, 전체 리드타임을 핵심 지표로 정의했습니다. 같은 기준으로 적용 전후를 비교해 실제 업무 변화와 다음 확장 여부를 판단하도록 설계했습니다.
03왜 전체 자동화가 아니라 3단계만 선택했나요?+
초기 단계는 입력 규칙을 정의하기 쉽고, 사용자 부담과 정보 누락이라는 핵심 문제에 직접 영향을 줍니다. 반면 생성·발행 단계까지 동시에 자동화하면 오류 원인과 효과를 구분하기 어렵습니다. 따라서 가장 큰 병목에 가까우면서 검증 가능한 범위를 먼저 골랐습니다.
04이 프로젝트에서 제가 맡은 판단과 기여는 무엇인가요?+
기존 프로세스 분석, 사용자별 문제 구분, TO-BE 흐름 설계, 자동화 우선순위 선정, 동적 요청 폼의 질문 구조와 검증 지표 정의를 맡았습니다. 비정형 요청을 실행 가능한 기준으로 바꾸고, 다음 단계의 판단 근거까지 설계한 것이 핵심 기여입니다.
